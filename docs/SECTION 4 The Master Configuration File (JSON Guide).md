### <span style="color: #4682B4;"><i class="fa-solid fa-book-open"></i> 4.1 Introduction to the Parametric Matrix</span>

While AeroForgeX v4.0 features a stunning Web GUI for rapid prototyping, the true industrial power of the software lies in its **JSON Configuration Matrix**. For automated server clusters, multi-node supercomputers, and professional aerospace design, graphical interfaces limit reproducibility and scriptability. 

A JSON file acts as the absolute DNA of your optimization run. It is highly readable, easily version-controlled via Git, and can be dynamically generated by external Python scripts for massive multi-stage design sweeps.

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-microchip"></i> 4.1.1 The AeroForgeX Parsing Engine </span>
To master AeroForgeX, you must understand how the internal Python parser interprets your JSON file. AeroForgeX has a very robust, fault-tolerant class, designed to minimize user errors:

<div style="background-color: #F8F9FA; border-left: 4px solid #708090; padding: 15px; border-radius: 4px; margin-top: 10px; margin-bottom: 20px;">
  <ul style="margin-bottom: 0px; padding-left: 20px; color: #444; line-height: 1.6;">
    <li><i class="fa-solid fa-text-height" style="color: #4682B4;"></i> <strong>Case Insensitivity:</strong> The parser forces all keys to lowercase internally. <code>"Opt_Opts"</code> is functionally identical to <code>"opt_opts"</code>.</li>
    <li><i class="fa-solid fa-broom" style="color: #2E8B57;"></i> <strong>Trailing Comma Auto-Fix:</strong> Standard JSON crashes if a trailing comma is left before a closing bracket <code>}</code>. AeroForgeX features a proprietary Regex auto-cleaner that seamlessly strips illegal commas, preventing a high-hour run from failing due to a typo.</li>
    <li><i class="fa-solid fa-life-ring" style="color: #FF8C00;"></i> <strong>Fallback Safety Injection:</strong> If a critical parameter is accidentally omitted, the parser safely injects an industry-standard fallback (e.g., defaulting to <code>161</code> panels), preventing <code>KeyError</code> crashes.</li>
    <li><i class="fa-solid fa-clock-rotate-left" style="color: #9370DB;"></i> <strong>Legacy Support:</strong> Still supports legacy <em>Xoptfoil2</em> Fortran <code>.txt</code> Namelists, though modern JSON is highly recommended. You can even use abbreviated keys (e.g., <code>"pop"</code>) or descriptive legacy keys (e.g., <code>"population_size"</code>).</li>
  </ul>
</div>

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-network-wired"></i> 4.1.2 The "Smart Array" Broadcasting System</span>
In multi-point optimizations, typing out values for 10 different operating points is tedious. AeroForgeX utilizes a **Smart Array Broadcaster**. If you specify `"num_pts": 5`, but only provide `"re_def": [500000.0]`, the parser automatically broadcasts that float to the remaining 4 points, dynamically building <code>[500k, 500k, 500k, 500k, 500k]</code> in RAM.

This block controls master inputs, outputs, and solver assignments. It defines the foundation of the computational environment.

  <span style="color: #0550AE;">"out_dir"</span>: <span style="color: #CF222E;">"Outputs"</span>,

*   <span style="color: #3B82F6; font-weight: bold;">`solver`:</span> (`"xfoil"` or `"neuralfoil"`) Routes geometry to Fortran (high accuracy) or CNN (high speed).
*   <span style="color: #3B82F6; font-weight: bold;">`shape_func`:</span> Mathematical sculptor (`"cst"`, `"bezier"`, `"hicks-henne"`, `"camb-thick"`).
*   <span style="color: #3B82F6; font-weight: bold;">`foil_file`:</span> Path to starting Seed Geometry. 
    <div style="background-color: #FFFBEB; border-left: 4px solid #F59E0B; padding: 8px 12px; margin-top: 5px; margin-bottom: 10px; color: #92400E; font-size: 0.9em; border-radius: 4px;">

*   <span style="color: #3B82F6; font-weight: bold;">`auto_dir`:</span> If `true`, system generates nested structures: `Outputs/[Prefix]/[Seed]/[Algo]/[Solver]/` to prevent batch collision.
*   <span style="color: #3B82F6; font-weight: bold;">`threads`:</span> `-1` commands AeroForgeX to utilize **100% of physical and logical CPU cores**. (Set to `4` or `8` to limit pool size).
*   <span style="color: #3B82F6; font-weight: bold;">`verbose`:</span> If `true`, prints detailed iteration physics to the terminal. Set to `false` for headless HPC runs to save I/O overhead.

### <span style="color: #4682B4;"><i class="fa-solid fa-cubes"></i> 4.3 Parametric Geometry Tuning Engines</span>
The AI will *only* parse the block corresponding to your chosen `"shape_func"`. The others will be safely ignored.

#### <span style="color: #1E90FF; font-weight: bold;">4.3.1 Kulfan CST (<code>cst_opts</code>)</span>
```json
"cst_opts": { "n_t": 6, "n_b": 6, "n1_t": 0.5, "n2_t": 1.0, "init_pert": 0.05 }
```
*   **`n_t` / `n_b`**: Number of Bernstein weights (Top/Bottom). `6` per side (12 vars) is standard. Going above `8` generates pressure-recovery micro-curves but requires massive populations.
*   **`n1` / `n2`**: Class Function exponents. `0.5` and `1.0` guarantee a perfectly round LE and sharp TE. Modify *only* for exotic re-entry vehicles.
*   **`init_pert`**: Swarm scatter size. `0.05` spawns Gen 0 within a tight $\pm 5\%$ boundary around the Seed Airfoil.

#### <span style="color: #1E90FF; font-weight: bold;">4.3.2 Bezier Polygons (<code>bez_opts</code>)</span>
```json
"bez_opts": { "ncp_t": 6, "ncp_b": 6, "init_pert": 0.10 }
```
*   **`ncp_t` / `ncp_b`**: Number of floating Control Points. Because Beziers define global curvature, fewer points (`5` to `7`) are needed. Ideal for wind turbine roots.

#### <span style="color: #1E90FF; font-weight: bold;">4.3.3 Hicks-Henne Sine Bumps (<code>hh_opts</code>)</span>
```json
"hh_opts": { "n_t": 5, "n_b": 3, "smooth": true, "init_pert": 0.05 }
```
*   **`n_t` / `n_b`**: Independent sine-wave bumps. Asymmetric configs (`5` Top, `3` Bottom) are highly recommended for Transonic Shockwave mitigation to focus CPU cycles on the upper surface.
*   **`smooth`**: **CRITICAL.** If `true`, runs a Bezier filter over the seed to "shrink-wrap" it into a flawless polynomial *before* bumps are applied, preventing wind-tunnel digitizing noise from crashing the solver.

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-code-branch"></i> 4.4.1 Core Algorithm Selection (<code>type</code> & <code>algo</code>)</span>
These parameters dictate the mathematical framework for the Stage 1 Global Evolutionary Search.

*   <span style="color: #3B82F6; font-weight: bold;">`type`:</span> Selects the master architecture:
    *   `1`: Standard Differential Evolution (DE) *(Natively powered by PyMoo)*.
    *   `2`: Custom jDE (Self-Adaptive Differential Evolution).
    *   `3`: Custom SHADE (Success-History Adaptive Differential Evolution).
*   <span style="color: #3B82F6; font-weight: bold;">`algo`:</span> Dictates the specific mutation strategy. Choose based on your problem complexity:

<div style="background-color: #F8F9FA; border-left: 4px solid #4682B4; padding: 15px; border-radius: 4px; margin-top: 10px; margin-bottom: 20px;">
  <ul style="margin-bottom: 0px; padding-left: 20px; font-size: 0.95em; color: #444; line-height: 1.6;">
    <li><strong style="color: #24292F;"><code>"de"</code> (Standard DE)</strong> - <span style="background-color: #E2E8F0; padding: 2px 6px; border-radius: 4px; font-size: 0.8em;">Type: 1</span><br> Uses fixed, unyielding mutation and crossover rates. Best for simple, low-dimensional problems like Camb-Thick scaling. Struggles heavily with coupled variables like CST due to fixed step sizes.</li>
    <li><strong style="color: #24292F;"><code>"jde"</code> (Self-Adaptive DE)</strong> - <span style="background-color: #E2E8F0; padding: 2px 6px; border-radius: 4px; font-size: 0.8em;">Type: 2</span><br> The safest "Daily Driver." Encodes mutation factors directly into the DNA of the airfoils. The algorithm literally learns which step sizes yield the best aerodynamics and shrinks its search radius as it approaches the optimum. </li>
    <li><strong style="color: #24292F;"><code>"shade"</code> (Success-History Adaptive)</strong> - <span style="background-color: #E2E8F0; padding: 2px 6px; border-radius: 4px; font-size: 0.8em;">Type: 3</span><br> The "Surgical Scalpel." Maintains a historical archive of successful mathematical leaps and uses Cauchy/Normal statistical distributions to generate new parameters. Pulls mutations toward the top 20% of the swarm. <strong>Absolute necessity for 12+ variables</strong> (e.g., High-Order Kulfan CST or Kinematic Flap co-optimization).</li>
    <li><strong style="color: #24292F;"><code>"lshade"</code> (Linear Pop. Reduction)</strong> - <span style="background-color: #E2E8F0; padding: 2px 6px; border-radius: 4px; font-size: 0.8em;">Type: 1</span><br> As the optimizer nears final generations, it dynamically shrinks the population size, violently forcing the algorithm from global exploration to aggressive local exploitation.</li>
    <li><strong style="color: #24292F;"><code>"nm-only"</code> (Nelder-Mead Only)</strong> - <span style="background-color: #E2E8F0; padding: 2px 6px; border-radius: 4px; font-size: 0.8em;">Type: 1</span><br> Skips evolution entirely. Use this if you have a 99% perfect airfoil and just want gradient-free local polishing to squeeze out the final $10^{-8}$ decimal points of drag.</li>
  </ul>
</div>

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-users-viewfinder"></i> 4.4.2 Swarm Sizing & Duration (<code>pop</code> & <code>gen</code>)</span>
These define your computational footprint and the duration of the run.

*   <span style="color: #3B82F6; font-weight: bold;">`pop`:</span> The number of unique airfoils evaluated simultaneously in a single generation.
*   <span style="color: #3B82F6; font-weight: bold;">`gen`:</span> Absolute termination limit. (Total evaluations = `pop` $\times$ `gen`). Once reached, Stage 1 terminates and hands off to Nelder-Mead polishing.

<div style="background: linear-gradient(135deg, #1E293B 0%, #0F172A 100%); border: 1px solid #FCD34D; padding: 20px; border-radius: 8px; color: #F8FAFC; margin-bottom: 20px; box-shadow: 0 4px 15px rgba(252, 211, 77, 0.15);">
  <h4 style="margin-top: 0px; color: #FCD34D; text-transform: uppercase; letter-spacing: 1px;"><i class="fa-solid fa-crown"></i> The Golden Rule of MDO</h4>
  <p style="font-size: 0.95em; line-height: 1.6; margin-bottom: 0px;">Your population size (<code>pop</code>) should ideally be at least <strong>$10 \times \text{Number of Design Variables}$</strong>. If you use 8 Top and 8 Bottom CST weights (16+2=18 variables), <code>pop: 180</code> is mathematically ideal to prevent genetic stagnation. <em>(Note: Thanks to AeroForgeX's efficient memetic algorithms, you can often safely reduce this to 40-60 for standard tasks to save time).</em></p>
</div>

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-sliders"></i> 4.4.3 Standard DE Hyperparameters (<code>f</code> & <code>cr</code>)</span>
*(Note: These are ONLY utilized if `"type": 1` is selected. jDE and SHADE calculate these internally).*

*   <span style="color: #3B82F6; font-weight: bold;">`f` (Mutation Factor):</span> Dictates the magnitude of the difference vector added to a base airfoil. A high number (`0.8`) causes wild, exploratory jumps. A low number (`0.3`) causes tight, exploitative local searches.
*   <span style="color: #3B82F6; font-weight: bold;">`cr` (Crossover Rate):</span> The probability that a mutated parameter replaces original DNA. `0.9` means heavy mixing between parent airfoils, ensuring rapid dissemination of good traits throughout the swarm.

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-user-shield"></i> 4.4.4 Constraint Resolution & Particle Mechanics</span>
Random mutations frequently generate mathematically impossible shapes. These settings handle those failures.

*   <span style="color: #3B82F6; font-weight: bold;">`rescue`:</span> **Highly Recommended.** If an airfoil mutates into an impossible shape, it "dies" against the boundary wall, receiving a penalty score of `1000.0`. If `true`, the system detects dead vectors, deletes them from RAM, and instantly *teleports* them to a new randomized valid coordinate, ensuring 100% of CPU power explores valid aerodynamics.
*   <span style="color: #3B82F6; font-weight: bold;">`retry`:</span> Before officially declaring a particle dead, the engine will slightly perturb the failing particle this many times to see if it can "nudge" it back into bounds.
*   <span style="color: #3B82F6; font-weight: bold;">`init_att`:</span> Generating Gen 0 via Latin Hypercube Sampling is difficult with strict constraints (e.g., demanding a perfectly 20% thick airfoil). The system attempts to randomly spawn shapes up to `1000` times until it fills the population with legally viable shapes.

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-gauge-high"></i> 4.4.5 Convergence Tolerances & Speed Limits</span>
Controls the physical limits of the AI's movement and early termination conditions.

*   <span style="color: #3B82F6; font-weight: bold;">`min_r` (Geometric Tolerance):</span> If the spatial volume of the entire swarm drops below this threshold (meaning every airfoil is mathematically identical up to the 4th decimal place), the AI assumes it has found the absolute bottom of the aerodynamic valley and terminates early to save hours of compute time.
*   <span style="color: #3B82F6; font-weight: bold;">`speed`:</span> The maximum velocity mutation vector a particle can travel in a single generation. Physically restricts chaotic `50%` coordinate jumps, preventing meshes from exploding into jagged noise.
*   <span style="color: #3B82F6; font-weight: bold;">`conv_prof`:</span> 
    *   `"exhaustive"`: Forces rigorous boundary-box search.
    *   `"quick_camb_thick"`: Auto-set when affine scaling is used; skips bounds-checking overhead to execute at maximum computational speed.

### <span style="color: #4682B4;"><i class="fa-solid fa-plane-departure"></i> 4.5 Operational Physics & Aerodynamics (<code>op_conds</code>)</span>

The `op_conds` (or `operating_conditions`) block is where you define the flight envelope. AeroForgeX is designed for **Multi-Point Optimization**, allowing you to pit conflicting physics against each other (e.g., optimizing for takeoff lift, cruise drag, and loiter endurance simultaneously). To achieve this, the block utilizes **Parallel Array Broadcasting**.

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-table-cells"></i> 4.5.1 The Matrix Blueprint</span>
*   <span style="color: #3B82F6; font-weight: bold;">`num_pts` (Integer):</span> **CRITICAL.** The total number of aerodynamic operating points the solver must evaluate for every single airfoil generation.
    <div style="background-color: #FEF2F2; border-left: 5px solid #EF4444; padding: 12px; border-radius: 4px; color: #991B1B; margin-top: 8px; margin-bottom: 12px; font-size: 0.95em;">
      <strong><i class="fa-solid fa-triangle-exclamation"></i> THE TRUNCATION TRAP:</strong> AeroForgeX strictly obeys this integer. If you set <code>"num_pts": 2</code>, but your arrays below contain 5 items, the system perfectly reads indices 1 and 2, and completely ignores indices 3, 4, and 5. This allows engineers to keep massive 15-point flight envelopes saved in their JSON, but easily dial <code>num_pts</code> down to run a rapid 5-minute feasibility test without deleting data.
    </div>
*   <span style="color: #3B82F6; font-weight: bold;">`re_def` / `ma_def` (Float):</span> Global scalar fallbacks. If an array is missing an index, the solver defaults to this Reynolds Number or Mach Number.

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-wind"></i> 4.5.2 The Aerodynamic State Vectors (Arrays)</span>
*   <span style="color: #3B82F6; font-weight: bold;">`mode` (Array of Strings):</span> Dictates how the physics solver attacks the operating point.
    *   `"spec-al"`: Specified Angle of Attack ($\alpha$). The solver locks the physical angle (e.g., $2.0^\circ$).
    *   `"spec-cl"`: Specified Lift Coefficient ($C_L$). **Advanced Feature:** The solver engages an internal Implicit Secant Root-Finder. It calculates the Lift Slope ($\frac{dC_L}{d\alpha}$) and iteratively adjusts the Angle of Attack until the airfoil generates the exact $C_L$ requested.
*   <span style="color: #3B82F6; font-weight: bold;">`val` (Array of Floats):</span> The numerical value corresponding to the mode. (`2.0` means $2.0^\circ$ $\alpha$, or $0.6$ $C_L$).
*   <span style="color: #3B82F6; font-weight: bold;">`re` / `ma` (Array of Floats):</span> The specific Reynolds Number and Mach Number for that exact flight regime.

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-square-root-variable"></i> 4.5.3 Mathematical Objective Functions (`opt_type`)</span>
Dictates the exact mathematical equation the PyMoo AI is trying to minimize.
*   <span style="background-color: #E2E8F0; padding: 2px 6px; border-radius: 4px; font-size: 0.85em; font-family: monospace;">"min-drag"</span>: Minimizes absolute profile drag ($C_D$). Used for high-speed cruise.
*   <span style="background-color: #E2E8F0; padding: 2px 6px; border-radius: 4px; font-size: 0.85em; font-family: monospace;">"max-glide"</span>: Maximizes the Glide Ratio ($L/D$). Used for climb performance and wind-turbine torque generation.
*   <span style="background-color: #E2E8F0; padding: 2px 6px; border-radius: 4px; font-size: 0.85em; font-family: monospace;">"min-sink"</span>: Maximizes Endurance Ratio ($C_L^{1.5} / C_D$). Used for thermal soaring and solar-powered UAV loiter time.
*   <span style="background-color: #E2E8F0; padding: 2px 6px; border-radius: 4px; font-size: 0.85em; font-family: monospace;">"max-lift"</span>: Maximizes absolute lift ($C_L$). Used for takeoff and landing approach stall mitigation.

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-crosshairs"></i> 4.5.4 Target Matching & Weighting</span>

If left unchecked, an aerodynamic AI optimizer will always seek the path of least mathematical resistance. ### <span style="color: #4682B4;"><i class="fa-solid fa-atom"></i> 4.9 Solver Physics & The SAP AI Filter (<code>xfoil_opts</code> & <code>panel_opts</code>)</span>

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-border-all"></i> 4.9.1 Mesh Generation (`panel_opts`)</span>


*   <span style="color: #3B82F6; font-weight: bold;">`le_clust` / `te_clust`:</span> AeroForgeX uses the **NASA Arc-Cosine clustering** mathematical transformation. `0.86` heavily concentrates panels at the leading edge. *Why?* Airflow accelerates massively around the nose, creating an extreme suction peak ($C_p$). Without dense paneling, the solver fails to resolve the peak, severely miscalculating lift and wave drag.

#### <span style="color: #1E90FF; font-weight: bold;"><i class="fa-solid fa-filter"></i> 4.9.3 The SAP Filter & Caching</span>
This is the most revolutionary performance upgrade in AeroForgeX v4.0. It acts as the ultimate aerodynamic filter, slashing total optimization times by up to **80%**.

```json
"sap_enabled": true, "nf_conf_threshold": 0.5, "cache_prec": 6, "parallel_op": true
```

*   <span style="color: #8A2BE2; font-weight: bold;">Surrogate-Assisted Pre-Screening (`sap_enabled`):</span> 
    During a global search, the AI generates thousands of disastrous airfoils before finding good ones. Sending a terrible airfoil to XFOIL wastes up to 12 seconds waiting for a Newton-Raphson timeout. If `true`, AeroForgeX wakes up the NeuralFoil Convolutional Neural Network (CNN) *before* sending a shape to Fortran. The CNN evaluates the airfoil in 5 milliseconds. If it predicts catastrophic separation, the shape is instantly killed. **XFOIL is never booted up for bad airfoils.**
*   <span style="color: #8A2BE2; font-weight: bold;">The Bayesian Lie-Detector (`nf_conf_threshold`):</span> 
    Deep Learning models suffer from "Out of Distribution" (OOD) hallucinations. If the optimizer mutates a bizarre, jagged, 40% thick rectangular wing, the AI has never seen it and will guess the drag completely wrong. NeuralFoil outputs a **Bayesian Uncertainty** metric. If set to `0.5`, any aerodynamic prediction where the AI is less than 50% confident is flagged as a hallucination. AeroForgeX rejects the AI's data and safely kills the shape, preventing the swarm from being tricked by fake physics.
*   <span style="color: #8A2BE2; font-weight: bold;">Vector-Hash Caching (`cache_prec`):</span> 
    Evolutionary algorithms frequently generate mathematically nearly identical airfoils twice, especially as the swarm converges in the Nelder-Mead stages. Running XFOIL twice on the same shape wastes CPU power. AeroForgeX uses an `OrderedDict` (LRU Cache). The `cache_prec` integer (`6`) aggressively rounds the Design Variables before hashing them. If an airfoil is mathematically identical to a previous one up to the 6th decimal place, the software instantly retrieves the saved Lift and Drag results directly from RAM.
